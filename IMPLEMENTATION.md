# IMPLEMENTATION.md — Внедрение SEO AI-автоматизации (BiXBiT)

> Human-facing playbook: как развернуть систему с нуля до полностью рабочего
> состояния. Это **руководство для оператора** (того, кто настраивает),
> в отличие от `RUNBOOK.md` (инструкция для Claude) и `README.md` (быстрый старт).
>
> Принцип всей системы: **не выдумываем данные, только белые методы,
> тех-факты о прошивках помечаем `[VERIFY]`.** Это руководство этому подчиняется.

| 0 | Предусловия: доступы, ключи, ПО | человек | 0.5–2 дня | нет |

## ФАЗА 0 — Предусловия (делает человек)

Ничего не настраиваем, пока не собрано **всё** из этого списка. Иначе Claude
застрянет на середине и придётся выдумывать (чего делать нельзя).

### 0.1 Доступы к аккаунтам
- [ ] Google Search Console — права на ресурс `bixbit.io` (минимум «Полный»).
- [ ] Google Analytics 4 — `GA4_PROPERTY_ID` и доступ к Data API.
- [ ] DataForSEO — активный аккаунт с балансом (логин/пароль).
- [ ] Google Cloud проект — для PageSpeed/CrUX API key и service account под GSC/GA4.
- [ ] Moz — бесплатный API-тир (Access ID + Secret Key).
- [ ] Bing Webmaster Tools — верифицированный сайт + API key.
- [ ] Доступ на запись в репозиторий сайта (для PR с правками мета/schema).

### 0.2 Программное обеспечение
- [ ] Node.js LTS (нужен для `npm i -g @anthropic-ai/claude-code` и DataForSEO MCP).
- [ ] Claude Code установлен и авторизован.
- [ ] Git настроен (для коммитов/PR).

### 0.3 Service account для Google API
1. В Google Cloud создать service account.
2. Скачать JSON-ключ.
3. Дать этому service account доступ в GSC (как пользователю) и в GA4 (как Viewer).
4. Включить в проекте: *Search Console API*, *Analytics Data API*, *PageSpeed Insights API*.

**Критерий выхода из фазы 0:** все чекбоксы выше отмечены, JSON-ключ скачан,
все ключи под рукой. Чего нет — фиксируем как «нет доступа», эти источники
система пометит как недоступные и не будет по ним выдумывать метрики.

## ФАЗА 1 — Каркас репозитория и `.env`

### 1.1 Развернуть структуру
Если структура из `README` ещё не создана, попросите Claude её собрать либо
создайте вручную:

```
.
├── CLAUDE.md
├── RUNBOOK.md
├── IMPLEMENTATION.md
├── .env.example
├── .mcp.json
├── .gitignore                 # .env обязательно в ignore!
├── .claude/
│   ├── agents/
│   ├── commands/
│   └── settings.json
├── .github/workflows/
├── data/
│   └── site-structure.md      # пустой шаблон — заполнится после первого аудита
├── content/
└── reports/
```

### 1.2 `.env.example` → `.env`
Шаблон `.env.example` (значения-плейсхолдеры, **реальные ключи только в `.env`**):

```bash
# DataForSEO
DATAFORSEO_LOGIN=your_login
DATAFORSEO_PASSWORD=your_password

# Google (service account JSON — путь к файлу вне репозитория)
GOOGLE_APPLICATION_CREDENTIALS=/secure/path/service-account.json
GSC_SITE_URL=https://bixbit.io/
GA4_PROPERTY_ID=000000000

# PageSpeed / CrUX
PAGESPEED_API_KEY=your_key

# Backlinks
MOZ_ACCESS_ID=your_id
MOZ_SECRET_KEY=your_secret
BING_WEBMASTER_API_KEY=your_key
```

```bash
cp .env.example .env   # заполнить реальными значениями
```

**Проверьте `.gitignore`:** `.env` и JSON service account **никогда** не должны
попасть в git.

**Критерий выхода:** структура создана, `.env` заполнен, секреты в `.gitignore`.

## ФАЗА 2 — Подключение источников данных

Это «кислород» системы. Подключаем и **сразу проверяем тестовым вызовом**.

### 2.1 DataForSEO MCP
```bash
claude mcp add dataforseo -- npx -y dataforseo-mcp-server
```
Или через `.mcp.json` (если предпочитаете версионировать конфиг — без секретов).

### 2.2 Проверка каждого источника
Запустите Claude в папке проекта и дайте задачу на верификацию (она же —
часть RUNBOOK §2):

```
> Прочитай RUNBOOK.md. Сделай быстрый тестовый вызов каждого источника
> (DataForSEO, GSC, GA4, PageSpeed, Moz, Bing) и выведи таблицу:
> источник | статус (OK/нет доступа) | что вернулось.
> Ничего не выдумывай — где нет доступа, так и пиши.
```

**Критерий выхода:** таблица доступности готова. Минимально для старта нужны
**GSC + DataForSEO**. Остальное желательно, но система отметит отсутствующее и
продолжит с тем, что есть.

## ФАЗА 3 — Сборка агентов и команд

`README`/`CLAUDE.md` ссылаются на 6 агентов и 6 команд — их нужно создать.
Делает Claude, человек ревьюит.

### 3.1 Агенты (`.claude/agents/`)
Попросить Claude сгенерировать по карте из `CLAUDE.md §8`:

| Файл | Зона |
|---|---|
| `technical-seo.md` | crawl, индексация, CWV, schema, sitemap, drift |
| `content-seo.md` | кластеры, брифы, E-E-A-T, контент-план |
| `geo-ai.md` | цитируемость в LLM, llms.txt, passage opt |
| `international-seo.md` | hreflang, локализация, мультирынки |
| `offpage-seo.md` | профиль ссылок, gap, мониторинг упоминаний |
| `local-seo.md` | NAP, GBP, дата-центры (Global Presence) |

Каждый агент должен наследовать жёсткие правила: реальные данные, `[VERIFY]`
на тех-факты, только белые методы, артефакты в нужные папки.

### 3.2 Команды (`.claude/commands/`)
По карте `CLAUDE.md §9`: `full-audit`, `page-optimize`, `new-cluster`,
`weekly-drift`, `competitor-gap`, `geo-check`. Каждая команда оркестрирует
нужных агентов и кладёт результат в `reports/` или `content/`.

### 3.3 `settings.json` — разрешения инструментов
Принцип белых методов и безопасности: разрешаем чтение/анализ и запись в
рабочие папки, **запрещаем** опасное. Пример:

```json
{
  "permissions": {
    "allow": [
      "Read", "Write", "Edit",
      "Bash(git diff:*)", "Bash(git status:*)",
      "WebFetch", "WebSearch"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Write(.env)",
      "Read(.env)"
    ]
  }
}
```
(`.env` читают скрипты/MCP, не сам Claude в чистом тексте — чтобы ключи не
утекли в логи/контекст.)

**Критерий выхода:** 6 агентов + 6 команд созданы, человек просмотрел их
тексты, `settings.json` не даёт деструктивных прав.

## ФАЗА 5 — Технический фундамент

Отрабатываем P0/P1 из аудита. Правки сайта идут **как diff/PR в репозиторий
сайта**, мёржит человек/CI (см. RUNBOOK §5).

Минимальный набор:
- Индексация и краул (robots, sitemap, канониклы, ошибки 4xx/5xx).
- Core Web Vitals: оптимизация 8k-рендеров (WebP/AVIF, lazy-load, preload LCP).
- JSON-LD: `Organization`, `Product`/`SoftwareApplication` (для firmware и
  cooling), `FAQ`, `Breadcrumb`, `Article` (блог).
- Регрессионная проверка мета/schema **перед каждым коммитом**.

**Критерий выхода:** все P0 закрыты или поставлены в PR; baseline обновлён;
ни одна правка не ушла в прод без diff-ревью человеком.

## ФАЗА 6 — Контент и GEO

- `/new-cluster` по топ-приоритетам (firmware Antminer/Whatsminer — первыми,
  см. `CLAUDE.md §3`).
- Создать `/llms.txt`, открыть AI-краулеров (агент `geo-ai`).
- `/geo-check` — снять baseline AI-видимости в ChatGPT/Perplexity/Claude.
- international-seo: hreflang-спека и план локализации.

**Все технические утверждения о прошивках (J/TH, °C, совместимость, тесты) —
строго `[VERIFY]`**, человек верифицирует перед публикацией. Это критично:
ложные тех-данные = репутационный риск.

**Критерий выхода:** ≥1 кластер с брифами и матрицей перелинковки в `content/`;
`llms.txt` опубликован; baseline GEO снят.

## ФАЗА 7 — Автоматизация

`.github/workflows/seo-automation.yml`:
- **Weekly drift** (`/weekly-drift`) — регрессии против baseline → алерт/issue.
- **Monthly content plan** — обновление контент-плана.
- **PR-регрессии** — проверка мета/schema на каждом PR в репозитории сайта.

Секреты — через GitHub Actions Secrets (не в коде). Расписание через `cron`.

**Критерий выхода:** workflow проходит на тестовом запуске; drift пишет отчёт
в `reports/`; алерты приходят туда, где их видит человек.

## Эксплуатация (после внедрения)

| Ритм | Действие |
|---|---|
| Еженедельно | drift-мониторинг, разбор алертов, проверка P0 |
| Ежемесячно | обновление контент-плана, `/geo-check`, ревью бэклинк-gap |
| Квартально | большой `/full-audit`, пересмотр приоритетов и стратегии |
| По событию | `/competitor-gap` при движениях Braiins/Vnish/LuxOS/ePIC |

Что система закрывает на 90–100% и что отдаёт человеку — см. `RUNBOOK.md §5`.
Стратегические/бюджетные решения, аутрич-переговоры, юр-нюансы и финальную
верификацию тех-фактов — всегда человек.

## Чего НЕ делать (риски внедрения)

1. Не запускать агентов до заполнения `data/site-structure.md` без пометок
   «предположение».
2. Не коммитить `.env` / service-account JSON.
3. Не публиковать `[VERIFY]`-факты без проверки человеком.
4. Никаких PBN, покупки ссылок, клоакинга, накрутки — только белые методы.
5. Не давать инвест-советов и обещаний доходности майнинга.
6. Не пускать правки сайта в прод без diff-ревью.
